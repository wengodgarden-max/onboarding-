## 一、故障原因分析

### 1.1 当前最紧迫故障：用户发消息返回 catch 块错误

**根因链：**

```
用户登录 → getUserInfo(userId, userName)
  → identity.ensureFresh() 可能抛异常（Excel 缺失/格式错误）
  → getUserInfo 返回 null 或直接崩溃
  → userInfo 为 null，后续所有 userInfo.xxx 访问抛出 TypeError
  → 最外层 catch 兜住，返回"抱歉出了点问题"
  → catch 块中 e.stack.split() 在某些错误类型下二次崩溃
  → terminal 无有效 stack trace → 开发者无法定位
```

**三个触发条件满足其一即崩溃：**
1. `userInfo` 为 null 时（用户不在花名册），任何 `userInfo.xxx` 访问
2. `/api/xhr/goal` 中未定义 `user` 变量的直接引用
3. catch 块本身 `e.stack.split()` 对异常类型不兼容

### 1.2 历史反复出现的同类 BUG 根因分析

| BUG 类型 | 出现次数 | 根因 | 根本矛盾 |
|----------|----------|------|----------|
| 变量未定义导致 ReferenceError | 3+ 次 | `/api/chat` 超过 300 行，`let`/`var` 混用，变量作用域横跨多个 if-else 分支 | 函数体过大 + 无分段错误隔离 |
| userInfo 空指针 | 2 次 | `getUserInfo` 无 try/catch，返回 null 后下游无防御 | 身份源不可靠 + 调用方无空值契约 |
| 目标制定数据丢失 | 持续 | `goal-engine` 和 `xhr-format` 字段契约不一致 | 模块间无 TypeScript/JSON Schema 约束 |
| 服务崩溃无日志 | 多次 | catch 块只打 message 不打 stack，二次错误覆盖原错误 | 错误处理设计缺陷 |
| 课程推荐泛滥 + 乱码 | 2 次 | 每次请求无条件跑推荐引擎，LLM 返回 URL-encoded 中文 | 推荐策略无"何时推荐"的约束 |

**系统性根因：**
- **缺少模块间接口契约**：模块 A 返回 `keyTasks`，模块 B 读 `tasks`——没有 JS Doc 或 TypeScript 保证
- **缺少分层错误隔离**：一个引擎崩溃不应该导致整个请求返回 500
- **缺少空值防御规范**：花名册数据不可靠，但下游代码假设它永远存在

## 七、长期预防策略

### 7.1 技术层面

1. **接口契约强制执行**：关键数据对象（goalData、userInfo、journeyNode）定义 JSDoc 类型注解，模块间传参统一字段名
2. **分层隔离原则**：每个引擎调用（journey/recommend/highlight/knowledge）包独立 try/catch，单引擎失败不影响全流程
3. **空值防御规范**：所有 `getXxx()` 函数必须返回 null 或空对象，调用方必须处理 null 情况
4. **错误日志规范**：catch 块必须 `console.error('[模块名]', e.message, '\n', e.stack)`，不打二次错误
5. **存储最小写原则**：哪个数据变了只写哪个文件，不用 saveAll()

### 7.2 流程层面

1. 每次改 `/api/chat` 后必须至少用两个账号（一个在册、一个不在册）做冒烟测试
2. 新增模块间数据传递时，必须检查上下游字段名是否一致
3. 改用户可见文案后，必须在浏览器里确认没有 mojibake
4. 发版前执行 `node --check *.js` 做语法检查
