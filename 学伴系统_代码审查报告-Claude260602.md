## P0 — 确认会崩的 Bug（必须先修）

### Bug 1：`/api/xhr/goal` 里的 `user` 变量未定义

**文件：** `server.js`，约 760 行附近

**现象：** 调用 `/api/xhr/goal` 时，如果走到 `else` 分支（没有传 `customData.kpis`），会执行这两行：

```javascript
goalData = xhrFormat.generateTrialGoalDraft(daysOnboard, profile.job || user.job || '', ...);
if (user) goalData.employeeName = user.name || '';
```

但是这个作用域里根本没有定义 `user`，`const user = store.getUser(userId)` 写在了别的路由里。这里会直接 `ReferenceError: user is not defined`，catch 不到，返回 500。

**修法：** 在这个路由里补一行：

```javascript
const user = store.getUser(userId) || {};
```

### Bug 2：`userInfo` 空指针——已知问题但兜底位置有误

**文件：** `server.js`，约 320 行

**现象：** `/api/chat` 里这样写的：

```javascript
var userInfo = getUserInfo(userId, user.name);
userInfo = userInfo || { name: user.name || '', job: '', ... };
```

表面上兜底了，但 `getIntentContext()` 函数（约 260 行）里做了这个：

```javascript
const job = userInfo.job || '员工';
const days = userInfo.daysOnboard || 0;
```

问题不在这里，而在 `userInfo` 被当作参数传进去之前，有多处直接读 `userInfo.xxx` 而没有判断 `userInfo` 是否存在。当花名册里查不到用户（新用户首次登录），`getUserInfo` 返回 `null`，兜底赋值是在 `getUserInfo` 调用之后做的，但是 `getIntentContext` 是在赋值之后调用的，所以这个兜底其实是有效的。

**真正的问题：** `buildGreeting` 里的 `userInfo` 参数在 `/api/topics/:userId` 路由（新建话题时）没有做兜底：

```javascript
var userInfo = getUserInfo(req.params.userId, null);
const greeting = buildGreeting(userInfo, !!userInfo);
```

`buildGreeting` 里第一个分支（`known === true`）会访问 `userInfo.name`、`userInfo.department` 等，但 `known` 是 `!!userInfo`，只有 `userInfo` 不为 null 时才进这个分支，所以这里逻辑上没问题。

**但实际上：** `identity.getStats()` 在花名册为空时返回 `conflicts: []`，这没问题，但 `getUserInfo` 里调用了 `identity.ensureFresh()`，如果 `identity.js` 本身加载失败（文件缺失、格式错误），会抛出异常，catch 不到，整个 `/api/chat` 就 500 了，而且 terminal 里也没有 stack trace（已知问题之一）。

**修法：** 在 `getUserInfo` 外层包一个 try/catch：

```javascript
function getUserInfo(userId, userName) {
  try {
    identity.ensureFresh();
    var info = identity.getIdentity(userId, userName);
    // ...
    return info;
  } catch (e) {
    console.error('[USER-INFO-ERROR]', e.message, e.stack);
    return null;
  }
}
```

### Bug 3：`knowledge_gaps.json` 直接 `fs.writeFileSync` 绕开了 `store`

**文件：** `server.js`，约 798、821、837 行

**现象：** 知识缺口的 gap 数组是用 `knowledgeSearch.getGaps()` 读出来的，然后直接 `fs.writeFileSync` 写回去，同时 `store.js` 的 `saveAll()` 里没有管这个文件。`knowledge-search.js` 里也是每次调用都 `loadGaps()`（同步读盘）。

如果两个请求同时修改同一个 gap，会产生写冲突，最后一个写操作覆盖前一个，数据丢失。

**修法（短期）：** 把 gap 的读写统一到 `knowledge-search.js` 内部，加一个内存缓存变量（类似 store 的做法），不要每次请求都读盘写盘：

```javascript
let _gaps = loadGaps(); // 启动时加载一次

function getGaps(status) {
  return status ? _gaps.filter(g => g.status === status) : _gaps;
}

function saveGaps() {
  fs.writeFileSync(..., JSON.stringify(_gaps, null, 2));
}
```

## P1 — 性能问题（高并发或数据量大时会明显）

### 问题 4：`store.js` 每次写操作都 `saveAll()` — 10 个文件同步写盘

**文件：** `store.js`

`saveAll()` 被调用了 10 次（每个写操作结束后）。它会同步 `writeFileSync` 写 6 个 JSON 文件（`users`、`topics`、`profiles`、`feedback`、`sessions`、`highlights`）。

单个用户发一条消息，`addMessage` → `saveAll()` → 写 6 个文件。如果 highlights 有 1000 条，`highlights.json` 可能有几 MB，每次都全量覆盖写。

`topics.json` 里存着所有人的全部对话历史，随着使用量增长会越来越大。

**修法：** 最小改动是「按文件分别保存」，只写变化的那一个：

```javascript
function saveUsers() { writeJSON('users.json', _users); }
function saveTopics() { writeJSON('topics.json', _topics); }
// ...

function addMessage(...) {
  // ...
  saveTopics(); // 只写 topics，不写其他 5 个
}
```

**进一步：** 加防抖，100ms 内多次写操作合并成一次：

```javascript
let _saveTimer = {};
function debounceSave(key, fn) {
  clearTimeout(_saveTimer[key]);
  _saveTimer[key] = setTimeout(fn, 100);
}
```

### 问题 5：`knowledge-search.js` 每次搜索都同步读 4 个 JSON 文件

**文件：** `knowledge-search.js`，约 99-102 行

`searchKnowledge()` 函数每次被调用都执行：

```javascript
const knowledgeEntries = loadKnowledgeEntries(); // fs.readFileSync
const highlights = loadHighlights();              // fs.readFileSync  
const profiles = loadProfiles();                  // fs.readFileSync
const topics = loadTopics();                      // fs.readFileSync
```

每条用户消息都会触发 `searchKnowledge`，每次都读 4 个文件。如果 `highlights.json` 很大，这是个明显的 I/O 瓶颈。

**修法：** 和 `store.js` 一样，改成内存缓存 + 按需刷新：

```javascript
let _entries = loadKnowledgeEntries();
let _entriesDirty = false;

function getAllEntries() { return _entries; }
function importEntry(entry) {
  // ...修改 _entries...
  _entriesDirty = true;
  saveKnowledgeEntries(_entries); // 保留同步写，但内存读不再反复读盘
}
```

### 问题 6：`/api/chat` 主流程过长，超过 300 行，错误处理粒度太粗

**文件：** `server.js`，约 290-600 行

整个 `/api/chat` 是一个巨大的 `async` 函数，最外层 try/catch 把所有错误都兜住，返回给前端一段错误文本。这导致：

- 哪一步出错不清楚（terminal 也可能看不到 stack）
- 课程推荐有一个单独的内层 try/catch，但旅程节点、闪光点检测等都没有保护
- `journeyEngine.shouldTriggerNode` 抛出异常会直接跳到最外层 catch，后面所有逻辑都不执行

**修法：** 把各个引擎的调用包成独立的 try/catch，允许单个引擎失败而不影响主流程：

```javascript
// 旅程检测
let journeyAlert = null;
try {
  const journeyNode = journeyEngine.shouldTriggerNode(userInfo, topic.lastJourneyTrigger);
  if (journeyNode) { journeyAlert = { ... }; }
} catch (e) {
  console.error('[JOURNEY-ERROR]', e.message);
}

// 闪光点检测
let newHighlights = [];
try {
  const hresult = highlightEngine.detectHighlights(message, history.slice(-6));
  if (hresult.hasHighlight) { ... }
} catch (e) {
  console.error('[HIGHLIGHT-ERROR]', e.message);
}
```

## P2 — 代码质量问题（不影响当前运行，但会给后续维护埋坑）

### 问题 7：`var` / `let` / `const` 混用，且部分 `var` 在 async 函数中

**文件：** `server.js`

`var` 存在变量提升，在 async 函数里容易出现意料之外的作用域行为。具体表现：`getUserInfo` 里用 `var info`，`/api/admin/roster/reload` 里用 `var s`，这些没问题，但主 `/api/chat` 里的 `var userInfo` 在多个 if/else 分支里被重新赋值，如果后面有人改代码可能引入提升 bug。

**修法：** 全面替换为 `const` / `let`，区分「只读」和「可变」：

```javascript
// 改前
var userInfo = getUserInfo(userId, user.name);
userInfo = userInfo || { ... };

// 改后
let userInfo = getUserInfo(userId, user.name);
if (!userInfo) userInfo = { ... };
```

### 问题 8：`journey.js` 里 D3 节点的 `proactiveMessage` 用了 Unicode 转义

**文件：** `journey.js`，约 70-90 行

D3 节点的 `proactiveMessage` 是一段拼接的数组，里面用了大量 `\u5165\u804c`（Unicode 转义），而其他节点都是直接写中文字符串。这不影响功能，但极难阅读和维护。

**修法：** 解码后替换为直接的中文字符串，和其他节点保持一致。

### 问题 9：`engine.js` 和 `brain.js` 存在功能重叠

**文件：** `engine.js` 和 `brain.js`

两个文件都定义了 `SYSTEM_PROMPT`，都有 `buildMessages()` 函数，`engine.js` 还定义了一套专门用于目标制定的 Prompt，但 `server.js` 里只 `require('./brain')`，没有引用 `engine.js`。

`engine.js` 的 `buildMessages` 函数目前是死代码。

**修法：** 确认 `engine.js` 是否还在使用。如果是早期遗留文件，删除或明确标注 `// DEPRECATED`。如果有用，在 `server.js` 里明确 require 并使用。

### 问题 10：管理后台的 gap 填写接口没有权限验证

**文件：** `server.js`，约 800-840 行

`/api/knowledge-gaps/:gapId/fill` 和 `/api/knowledge-gaps/:gapId/assign` 这两个接口直接修改知识库内容，但没有任何身份验证（无 token、无角色校验）。任何知道接口地址的人都可以向知识库写入内容。

**修法（最小成本）：** 加一个简单的管理员 token 校验：

```javascript
function requireAdmin(req, res, next) {
  const token = req.headers['x-admin-token'] || req.query.token;
  if (token !== config.adminToken) {
    return res.status(401).json({ error: '需要管理员权限' });
  }
  next();
}

// 在路由上挂载
app.post('/api/knowledge-gaps/:gapId/fill', requireAdmin, (req, res) => { ... });
```

在 `config.json` 里加 `"adminToken": "你设置的token"`。

## 附：已知问题的补充分析

**「部分用户发消息返回 catch 块错误，terminal 无 stack trace」**

根因是最外层 catch 里这样写的：

```javascript
} catch (e) {
  console.error('[CHAT-ERROR]', e.message);  // 只打了 message，没打 stack
  res.json({ reply: '❌ 出错了...' + e.message + '\n' + e.stack.split('\n').slice(0,1) });
}
```

`e.stack` 在某些情况下可能是 `undefined`（比如某些 native 错误），`e.stack.split()` 就会抛出二次错误，导致 terminal 只看到 `[CHAT-ERROR]` 而没有位置信息。

**修法：**

```javascript
} catch (e) {
  console.error('[CHAT-ERROR]', e.message, '\n', e.stack || '(no stack)');
  res.json({
    reply: '❌ 出错了，请反馈：' + e.message,
    profile: null, recs: [], questions: [], highlights: [], engineOnly: true,
  });
}
```

不要把 `e.stack` 暴露给前端用户，只打到 terminal 里。
